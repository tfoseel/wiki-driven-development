# WDD Harness + Mini Booking Pilot Design

Date: 2026-06-29

## Purpose

This repository will become an open-source Wiki-Driven Development harness for Next.js projects.

The core premise is:

> The wiki is the SSOT. Code is derived from the wiki, but authored by agents, not generated by a deterministic code generator.

The harness must make the wiki-first cadence enforceable:

1. Edit the relevant wiki page first.
2. Compute the impacted wiki pages.
3. Resolve the code files owned or referenced by those wiki pages.
4. Modify only that impacted code surface.
5. Verify the gates declared by the wiki.
6. Mark stale or drifted outputs so the next agent session can continue safely.

The first pilot will live inside this repository as a small but complete booking app.

## Non-Goals

- Do not build a full automatic Next.js code generator.
- Do not start with a large 50-screen pilot.
- Do not require a specific AI agent vendor.
- Do not make the wiki viewer the first milestone.
- Do not hide product logic in code-only conventions.

The harness should be deterministic about graph, impact, context, and verification. The application code remains agent-authored.

## Harness And Pilot Relationship

This repository has two layers:

1. **WDD harness layer**: workflow skills, wiki templates, lint scripts, graph/index tooling, impact/session/drift/verify commands.
2. **Pilot app layer**: a small booking app whose wiki and Next.js code are created through the WDD harness.

The pilot should not have special rules that bypass the harness. If the pilot needs to change booking cancellation behavior, the agent should use the same cadence expected of downstream adopters:

```txt
edit pilot/wiki/actions/cancel-booking.md
→ run wdd impact pilot/wiki actions/cancel-booking
→ read impacted wiki pages
→ edit only referenced pilot/app files
→ run wdd verify / drift
```

This makes the pilot both an example application and a regression test for the WDD workflow itself.

## Wiki Node Model

A node is one markdown page under `wiki/**/*.md` or `pilot/wiki/**/*.md`.

Each node has frontmatter with at least:

```yaml
id: pages/booking-new
type: page
title: New Booking
summary: Select a service, date, slot, and contact details.
depends_on: []
implemented_by: []
verified_by: []
```

The important rule:

> If a concept needs impact tracking, it must become a wiki page.

For example, "cancel a booking" is not just a bullet inside a page. It is an `action` node, because it owns input validation, mutation rules, affected pages, and QA gates.

## Wiki Readability

The wiki must remain a product document, not a YAML database.

Frontmatter is only the quiet machine-readable layer for graph and ownership. The body is the human-readable SSOT for product logic. A planner should be able to fold the frontmatter and still understand the feature, behavior, policy, states, and QA expectations.

Recommended body shape:

- `page`: description, conditions, user actions, visible states/exceptions, navigation/handoff, independent QA.
- `action`: intent, input, rules, state changes, failure cases, QA.
- `model`: domain meaning, fields, validation, examples, mapping to entities.
- `entity`: domain meaning first, then persistence fields, constraints, migration notes.
- `flow`: start/end, handoff data, cross-page assertions.
- `qa`: product scenarios in given/when/then form.

Derived information such as impacted pages should be shown by `wdd impact` or a viewer, not authored into the prose.

Initial node types:

- `entity`: persistent data contract and migration expectations.
- `model`: domain shape, validation, entity mapping.
- `action`: write API contract, usually a Next.js server action.
- `page`: user-facing route, read model, visible states, page-level QA.
- `flow`: multi-page navigation and handoff contract.
- `policy`: cross-cutting business rule.
- `qa`: executable scenario set.
- `term`: reusable domain language.
- `design`: tokens and visual contract.

## Code Ownership Fields

Wiki pages should point to code, but code should not become the SSOT for product meaning.

Common fields:

```yaml
implemented_by:
  - pilot/app/src/actions/create-booking.ts
verified_by:
  - pilot/app/tests/e2e/create-booking.spec.ts
artifacts:
  - pilot/app/screens/booking-new/screen.tsx
```

The harness may also infer conventional files, but explicit references win. This lets agents answer: "This wiki page changed; what code surface is allowed to move?"

## Declared vs Derived Impact

Wiki pages should contain enough metadata for the harness to derive impact, but they should not manually maintain a full "all pages affected by me" list. That list is a backreference and will drift if people edit it by hand.

Authored fields:

```yaml
depends_on:
  - models/booking
  - entities/bookings
policies:
  - policies/cancellation-policy
implemented_by:
  - pilot/app/src/actions/cancel-booking.ts
verified_by:
  - pilot/app/tests/e2e/cancel-booking.spec.ts
artifacts:
  - pilot/app/src/screens/booking-detail/screen.tsx
```

Derived fields:

```txt
impacted_pages
upstream_dependencies
downstream_dependents
affected_code_files
missing_code_files
stale_code_files
```

The derived fields can be printed by `wdd impact`, stored in an index artifact, or shown in a viewer, but they should not be the authored SSOT inside the wiki page.

This keeps the useful part in the wiki: what this page depends on, what code it owns, and what verifies it. The harness computes the reverse impact set.

## Harness Commands

The MVP command set is orchestration-focused:

```txt
wdd index [wikiRoot]       # Parse wiki pages into a node graph.
wdd impact <node>          # Print impacted wiki nodes and owned code files.
wdd session <node>         # Print an agent-ready context pack.
wdd drift                  # Detect wiki/code stale candidates.
wdd verify <node>          # Run verification commands declared by impacted nodes.
```

`wdd session` is the agent handoff primitive. It should include:

- The selected node.
- Upstream dependencies.
- Downstream dependents.
- Referenced code files.
- Required gates.
- The wiki-first cadence reminder.

## Impact Graph

The graph has three overlapping layers:

- Ownership tree: feature/page grouping for human navigation.
- Contract DAG: `depends_on`, `reads`, `uses`, `actions`, `models`, `entities`, `policies`.
- Navigation graph: page-to-page and flow edges.

Impact should be conservative. If `actions/cancel-booking` changes, likely impacted nodes include:

- `models/booking`
- `entities/bookings`
- `pages/booking-detail`
- `flows/manage-booking-flow`
- `policies/cancellation-policy`
- `qa/cancel-booking-e2e`
- All referenced implementation and test files.

Conservative impact is acceptable because agents can clear non-real impact after reviewing the affected wiki pages.

## Agent Cadence

`AGENTS.md` should make this cadence mandatory for product or domain changes:

1. Locate or create the relevant wiki node.
2. Edit the wiki first.
3. Run `wdd impact <node>`.
4. Read impacted wiki pages before editing code.
5. Edit only code owned or referenced by impacted pages.
6. Run `wdd verify <node>` or the listed verification commands.
7. Run `wdd drift` and resolve or document remaining stale surfaces.

Infrastructure-only changes may use a tooling/process node instead of a product node.

## Mini Booking Pilot

The pilot should be small, complete, and able to exercise every important WDD node type.

Domain: a consultation booking app.

User journey:

1. View available services.
2. Start a booking for a service.
3. Choose a date and time slot.
4. Enter customer details.
5. Create the booking.
6. View booking details.
7. Reschedule or cancel according to policy.

## Booking State Model

Initial booking statuses:

- `draft`: user is composing a booking.
- `confirmed`: booking has been created and owns a slot.
- `rescheduled`: booking moved to a different slot.
- `cancelled`: booking released its slot and is no longer active.

Initial slot statuses:

- `available`
- `booked`
- `unavailable`

The policy node defines when reschedule and cancellation are allowed.

## Error Handling

Harness errors:

- Missing frontmatter or duplicate node IDs fail `wdd index`.
- Dangling `depends_on` links fail `wdd index`.
- Missing referenced code files warn during `wdd impact` until implementation exists.
- Missing verification files warn before a node is marked complete.
- Code changes outside the impact set should be reported by `wdd drift` once file ownership is indexed.

Pilot app errors:

- Slot no longer available.
- Booking not found.
- Cancellation not allowed by policy.
- Reschedule target slot unavailable.
- Invalid customer contact details.

Each app error should have a wiki-owned state and QA assertion.

## QA Coverage Model

QA nodes and page/action QA sections should partition the product input space instead of collecting a loose list of examples.

Every meaningful workflow should cover:

- Happy path.
- Empty or unavailable states.
- Policy boundaries.
- Validation failures.
- Mutation failure cases.
- Duplicate or repeated user actions.
- Stale data or race-like cases.
- Missing or invalid route/handoff data.
- Server/network errors.

For the mini booking pilot, this means at least:

- No available slots.
- Slot becomes booked after the user selected it.
- Invalid customer contact details.
- Create/reschedule/cancel button double-click.
- Booking id missing or unknown.
- Already cancelled booking is cancelled again.
- Cancellation is attempted inside the 24-hour policy boundary.
- Reschedule target slot is unavailable.
- Service is inactive.
- Timezone/date boundary around the cancellation cutoff.

## Verification Strategy

Harness tests:

- Parse valid and invalid node frontmatter.
- Build graph edges from `depends_on` and wiki links.
- Compute impact from entity, action, page, policy, and QA nodes.
- Generate stable `wdd session` output.
- Resolve `implemented_by` and `verified_by` files.

Pilot verification:

- Unit tests for model validation.
- Server action tests for booking creation, reschedule, and cancellation.
- Playwright E2E for create, reschedule, cancel, empty/unavailable states.
- Screenshot capture can come after the main WDD loop is working.

## Acceptance Criteria

The design is implemented enough when:

- A user can edit `pilot/wiki/actions/cancel-booking.md`.
- `wdd impact actions/cancel-booking` lists impacted wiki nodes and code files.
- `wdd session actions/cancel-booking` produces a useful agent context pack.
- An agent can update the wiki first, then the impacted Next.js code.
- `wdd verify actions/cancel-booking` runs the declared checks.
- `wdd drift` can identify stale or missing implementation surfaces.

This proves WDD as a wiki-aware agent runtime rather than a static documentation convention.
