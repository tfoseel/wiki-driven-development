상태: ✅ 검증 완료 · 코드 불필요 · 검증 불필요

## 목적
이 파일럿은 작지만 완결된 제품을 위키 문서를 SSOT로 삼아 운영할 수 있는지 검증한다.

## 하네스와의 관계
이 파일럿 위키는 특수한 예제가 아니라 WDD 하네스가 검증하는 제품 위키 샘플이다. 다른 앱도 Next.js 기본 구조 위에 `wiki/`, `harness/`, `AGENTS.md`를 얹고 위키 노드의 ownership metadata를 채우면 같은 린트와 ready 게이트를 사용할 수 있어야 한다.

## 읽기 구조
- 위키의 기본 읽기 표면은 GitHub Markdown이다.
- HTML 렌더링 산출물은 코어가 아니라 선택적 파생물이다.
- `wiki/` 안의 Markdown 원문이 제품 SSOT다.
- `wiki/` 안의 폴더명과 파일명은 사람이 읽기 쉬운 한국어를 사용한다.
- `wiki/QA/`는 흐름을 반복 설명하지 않고 실행 가능한 커버리지와 엣지 케이스를 기록한다.
- 안정적인 기계 참조는 파일 경로가 아니라 숨김 WDD 메타의 `id`를 사용한다.
- `wdd_status`는 기계가 읽는 상태 truth다.
- `## 상태`의 첫 줄은 사람이 읽는 canonical 요약이며 하네스가 `wdd_status`와 일치하는지 검증한다.
- 영향 범위, 구현 파일, 검증 파일, 실행 명령은 `<details>` 접이식 메타 안에서 읽는다.

## 운영 cadence
1. 사용자는 GitHub Markdown으로 화면, 정책, 액션, QA 상태를 확인한다.
2. 에이전트는 위키 문서를 먼저 수정하고 영향받는 위키 노드와 코드 파일을 추적한다.
3. 에이전트는 참조된 코드를 수정한다.
4. 에이전트는 E2E, QA 스크린샷, 흐름 화면 트리 캡처를 갱신한다.
5. 검증이 끝나면 에이전트는 `wdd_status`를 검증 완료 상태로 유지한다.
6. `wdd` 명령은 사용자가 외워야 하는 인터페이스가 아니라 에이전트와 CI가 사용하는 하네스다.

## Markdown 문서가 보여야 하는 것
- 제목과 요약은 제품 의미를 먼저 설명한다.
- 상태 줄은 이모지와 짧은 문구로 현재 phase, 코드 반영, 검증 상태를 보여준다.
- 구현 메타는 읽기 흐름을 방해하지 않도록 접힌 영역에 둔다.
- 본문은 기획자가 제품 로직을 이해할 수 있는 산문, 표, QA 시나리오, Mermaid 다이어그램으로 작성한다.
- QA가 완료된 화면 노드는 화면 스크린샷을 본문 가까이에 둔다.
- 흐름 노드는 생성된 화면 트리 캡처를 먼저 보여주고, 접힌 Mermaid 소스로 주요 분기와 의존 화면의 실제 스크린샷 참조를 남긴다.

## 작업 추적
- 제품 위키는 현재 검증된 제품의 SSOT다.
- 새 요청, 병렬 작업 상태, 리뷰 대화는 GitHub Issue와 PR에서 추적한다.
- 에이전트는 이슈를 픽업한 뒤 제품 위키를 먼저 수정하고, 코드와 QA를 동기화한 뒤 PR로 검증 증거를 남긴다.
- PR이 병합되기 전까지 제품 위키의 현재 브랜치가 해당 변경의 truth다.

## 사용자 여정
1. 예약 가능한 서비스를 본다.
2. 예약을 시작한다.
3. 날짜와 시간 슬롯을 고른다.
4. 고객 정보를 입력한다.
5. 예약을 생성한다.
6. 예약 상세를 확인한다.
7. 정책에 따라 일정을 변경하거나 취소한다.
