# Screen Nodes

## When to use

Use this for route-backed UI screens and visible states.

## Authoring contract

- Name route, purpose, read models, actions, empty/error/loading states, and user controls.
- Keep screenshot metadata route-backed once code is reflected.
- Embed the latest QA screenshot in the body.
- Keep mockups only as planning evidence; replace or compare with shipped screenshots during verification.
- Link flows and QA nodes that cover the screen.

## Evidence

Screen changes require visual evidence from `screenshots.route` and E2E or component tests for expected behavior.
